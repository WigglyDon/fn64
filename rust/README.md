# fn64 Machine Workspace

This Cargo workspace is fn64's sole current product implementation. Rust is
implementation material rather than fn64's product identity, and the current
machine remains deliberately incomplete and headless.

## Ownership

`fn64-core` owns represented emulated truth. A `Machine` instance owns its
cartridge, CPU, represented COP0 state, RDRAM, SP DMEM, reservation state, and
lifecycle. Core code must not own file paths, CLI parsing, windows, input,
audio-device plumbing, presentation, platform event loops, or platform-clock
stepping.

`fn64-inspection` owns only deterministic synthetic setup, assertions, plain
output, and process exit for the no-window probes. It consumes public core
surfaces and never owns machine state or runtime policy.

The single detailed owner for current represented capability and explicit
absence is [the represented-machine capability ledger](PARITY.md). Stable
architecture boundaries are indexed from [docs/INDEX.md](../docs/INDEX.md), and
the retired implementation era is recorded in
[PROJECT_HISTORY.md](../docs/context/PROJECT_HISTORY.md).

## Layout law

Modules name concrete machine owners:

- `cartridge`, `cartridge::byte_order`, and `cartridge::metadata`
- `machine` and `machine::rdram_reservation`
- `cpu`, `cpu::address`, `cpu::cop0`, `cpu::instruction`,
  `cpu::registers`, and `cpu::scalars`
- `rdram`
- `sp_dmem`

Do not add broad buckets such as `util`, `common`, `misc`, `helpers`, `engine`,
`platform`, `graphics_api`, or `emulator`. Add a module only when a specific
fn64 truth owner has been earned.

## Capability boundary

The workspace represents a bounded machine construction/reset, storage,
instruction classification, and straight-line `Machine::step` subset. It does
not claim complete CPU execution, cartridge boot, PIF/BIOS behavior, full
device or memory routing, timing accuracy, game compatibility, or a
window/audio host.

Do not infer public step support from a decoded instruction identity, a private
helper, or a green unit test. Capability changes must update the detailed
ledger at the same product commit.

## Verification

From the repository root, run:

```sh
./rust/verify-forward
```

That executable is the sole owner of the required command order. It runs the
format, warnings-denied lint, complete test, construction/reset-probe, and
represented-step-probe stages and ends with `forward gate: ok` on success.

## Repository policy

`Cargo.lock` is tracked for reproducible local verification. The workspace is
`publish = false`, and `rust/target` is ignored generated output.

Tests and probes use generated byte arrays and instruction words only. Do not
add commercial ROMs, proprietary BIOS/PIF blobs, copyrighted fixtures, or
circumvention material.
