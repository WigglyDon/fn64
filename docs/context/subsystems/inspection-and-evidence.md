# Inspection And Evidence

Context role: no-window proof and evidence context.
Scope: `fn64-inspection`, probe artifacts, and evidence lineage.
Canonical for: proof-shell ownership and proof limitations.
Not canonical for: machine behavior or release acceptance.
Inherits: [root law](../../../AGENTS.md) and [inspection scope law](../../../rust/crates/fn64-inspection/AGENTS.md).
Current-state owner: [CURRENT_STATE.md](../CURRENT_STATE.md).
Related evidence: [evidence process](../../process/EVIDENCE_AND_RELEASE.md) and `ops/evidence/` schemas.
Update triggers: probe purpose, output contract, evidence class, or manifest ownership changes.

`fn64-inspection` owns deterministic synthetic setup through public machine APIs,
plain formatting, assertion policy, and process exit. `fn64-core` owns every
emulated fact. Probe-specific policy must not move into the core.

Accepted evidence follows `cause → lawful generated bytes/input → named address/subsystem → Machine action/state mutation → observable result → artifact digest`.
Every artifact names source SHA, Context-SHA, command, working directory, exit
status, and unavailable facts. Similar text output is not semantic equivalence.

`fn64_machine_probe` proves construction/reset only. `fn64_step_probe` calls
public `Machine::step` for its eight represented cases and ends deterministically.
`fn64_boot_probe` is a separate bounded ROM-path inspection shell: it reads one
authorized local file, passes owned bytes into public core APIs, and reports
Machine-owned provenance, mutation lineage, and the first frontier. The
accepted run proves BOOT-2 only—one `SpecialAdd` commit followed by an
aligned `Lw` that rejects before mutation because SP IMEM offset zero is
unknown. None proves BOOT-3, full ISA, timing, window/audio/runtime,
performance, or game compatibility.

Inspection may depend on public core APIs. Core-to-inspection, private seam
calls, mutable CPU/COP0 backdoors, host event loops, and proprietary assets are
forbidden. Rollback and observability claims must cite the exact case.

An explicit user-supplied PIF firmware path is authorized as a future bounded
host input. Inspection may read only a literal selected path, report failure,
and transfer owned bytes. It must not search, download, bundle, reconstruct,
dump, or select game behavior; Machine validation and state production remain
core authority. No such input support exists yet.

Required validation: `./rust/verify-forward`; the boot probe and private-input
digest/size are separate explicit evidence. Evidence manifests additionally use
`tools/fleet/evidence-manifest check`. Graphical host and performance
observations remain explicitly unavailable.
